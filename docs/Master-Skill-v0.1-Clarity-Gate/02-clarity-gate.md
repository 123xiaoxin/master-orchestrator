# Requirement Clarity Gate

The clarity gate decides whether Master should execute, clarify, or produce a minimum prototype.

## What The Gate Measures

The gate measures execution clarity, not conversation count.

Master should evaluate:

- whether the real goal is identifiable
- whether task boundaries are explicit enough
- whether allowed and forbidden actions are known
- whether missing information blocks the next useful action
- whether risks are acceptable for a minimum step
- whether validation points can be stated

## 75%-80% Execution Clarity

When a task reaches roughly 75%-80% execution clarity, Master should stop broad clarification and produce a minimum prototype. More questions at that point often delay learning.

75%-80% is a heuristic judgment range, not a mathematical hard threshold. Master should combine the real goal, key variables, risk boundaries, and minimum prototype feasibility before deciding.

The threshold means:

- the real goal is clear enough
- the next artifact type is clear enough
- major risks are named
- remaining uncertainty can be tested through a small artifact
- the user can confirm or correct direction after seeing the prototype

## No Fixed Clarification Rounds

Master must not use "3-5 rounds" as a fixed ritual. Some tasks need zero clarification. Some need one precise question. Some need a prototype. Some need a pause during execution.

The correct question is not "Have we asked enough rounds?" It is "Can we safely produce the next useful contract or artifact?"

## Three Decisions

| Decision | When to choose it | Output |
|----------|-------------------|--------|
| `execute` | Goal, boundaries, actions, and validation are clear enough | Execution contract and action |
| `clarify` | A missing variable blocks safe next action | One or more targeted questions |
| `prototype` | Direction is likely enough but needs fast confirmation | Minimum prototype plus assumptions |

The gate should prefer the smallest useful next move.
