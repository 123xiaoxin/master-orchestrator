# Minimum Prototype

A minimum prototype is a small artifact used to confirm direction before larger execution. It is not a final delivery and not a substitute for the execution contract.

## Why Prototype Instead Of More Questions

When the requirement is roughly 75%-80% clear, more broad questions usually create friction. A small artifact makes the user's implicit preferences visible faster.

75%-80% is a heuristic judgment range, not a mathematical hard threshold. Master should combine the real goal, key variables, risk boundaries, and minimum prototype feasibility before deciding.

The prototype should:

- expose assumptions
- make direction concrete
- reduce abstract debate
- reveal missing variables
- give the user something easy to confirm or reject

## Prototype Types

| Task type | Minimum prototype |
|-----------|-------------------|
| Writing or documentation | Document outline |
| Product or interface | UI sketch or screen structure |
| Workflow or operations | Flow diagram |
| Prompt/system behavior | Prompt draft |
| Software feature | Minimum code skeleton |
| Multi-step execution | Execution contract |

## Prototype Requirements

A good minimum prototype is:

- small enough to revise cheaply
- concrete enough to reveal misunderstanding
- tied to the real goal
- explicit about assumptions
- paired with validation points

## Anti-Patterns

- turning a prototype into a full build
- using a prototype to avoid naming risk
- asking more broad questions when a draft would clarify faster
- creating an Agent just to produce a trivial prototype
