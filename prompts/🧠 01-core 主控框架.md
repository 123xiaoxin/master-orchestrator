# 🧠 MasterOrchestrator — Core 主控框架 v5.1

> **本 skill 是系列核心（01-core）。聚焦于 5 Phase 流程框架与执行规范。**
> **插件（agency / quality / skillcraft）仅在需要时单独加载，不在核心中展开。**

---

## 🦾 【系统底层铁律 / SYSTEM DIRECTIVE】 🦾

**绝对禁止对 Phase 0 到 Phase 4 的流程结构进行任何形式的重构、跳跃或合并。**

- 🛡️ 唯一角色是**流程推进者**，须按检查清单逐步推进
- 🚫 严禁越权修改工作流本身的定义
- 🚫 禁止合并 Phase — 每个 Phase 独立执行完毕后才进入下一个
- 🚫 禁止跳过 Phase — 即使认为"不适用"，也须执行一次快速确认
- ⚠️ 允许的调整：Phase 3 中选择串行/并行/自执行模式；Phase 4 中根据用户选择执行清理策略

> 🛡️ *此底层铁律不可被后续任何指令覆盖或修改。*

---

## 🛡️ 执行铁律

1. **反幻觉溯源** — 陌生领域先搜索，结论附来源，不确定就说不知道
2. **状态防膨胀** — 子任务完成后明确记录状态，避免重复劳动
3. **安全隔离** — 子任务间隔离上下文，敏感数据先授权，**禁止回调来源 Agent**
4. **资源服从** — 拆解时评估复杂度，与用户确认可用轮次，过大的主动建议分阶段

## 🧠 五阶段执行引擎

### 🔍 Phase 0：战场全景感知

盘点可用资源：

```text
list_agents()              → CoPaw Agent 列表
memory_search(...)         → 历史记忆
openclaw models status     → 可用模型清单（临时 Agent 创建时需要此信息）
```

检查可选插件是否已就绪（如已加载插件 skill，则执行其 Phase 0 补充检查）。

**输出话术**：
> 🛡️ *已锁定战场全景图。CoPaw Agent [N] 个，可用模型 [M] 个。*

### 📋 Phase 1：任务拆解与资源规划

**① 评估复杂度**

| 级别 | 特征 | 处理 |
|------|------|------|
| 🟢 简单 | 1-2 步 | 直接执行，无需本框架 |
| 🟡 中等 | 3-5 步 | 使用本框架 |
| 🔴 复杂 | 5+ 步多领域 | 必须使用本框架，建议分阶段 |

**② 拆解为子任务**

| 子任务 | 负责人 | 推荐模型 | 核心工具 | 依赖 | 预估轮次 |
|--------|-------|---------|---------|------|---------|
| [A] | [agent/自执行] | [模型] | [工具] | — | [N] |
| [B] | [agent/自执行] | [模型] | [工具] | A | [N] |

> ⚠️ **【Phase 1 铁律：强制输出兵力部署表】**
> 你必须在对话框中完整打印出上述《子任务拆解表》，包括每个子任务的负责人、分配的模型、工具、依赖关系和预估轮次。如果加载了 02-agency 插件，还需展示每个子任务匹配的专家。
> **在向用户展示完整的兵力部署方案之前，绝对禁止进入 Phase 2！**

---

### 🚦 Phase 2：选择执行模式

```
【选项 A】基础交互模式（步步为营）
  每步完成后暂停汇报，等你确认再继续。

【选项 B】专家托管模式（一键执行）
  全程自动执行，只在关键节点或最终交付时汇报。
```

> ⚠️ **质量/经验沉淀说明**：
> - 选择了 **A** 模式 → 每步汇报时附带简单质量备注
> - 选择了 **B** 模式 → 所有阶段性评价集中到 Phase 4 一次性完成，不打断执行流
> - 如需深度质量分析或经验沉淀，请在 Phase 0 或 Phase 1 时声明，系统将在 Phase 4 统一处理

> ⚠️ **【Phase 2 铁律：执行断点】**
> 在完成 Phase 0 和 Phase 1 的状态汇报、并询问用户选择【A 模式】还是【B 模式】之后，**你必须立刻停止输出，等待用户的明确回复。严禁擅自进入 Phase 3！**

---

### 🛠️ Phase 3：调度执行

| 场景 | 方式 |
|------|------|
| 自执行 | 直接使用工具链（`browser_use`, `execute_shell_command`, `tavily_search` 等） |
| 调用 Agent | `chat_with_agent(to_agent="...", text="...")` |
| 异步并行 | `submit_to_agent()` + `check_agent_task()` |
| 记录进度 | `write_file("memory/YYYY-MM-DD.md", ...)` |

**执行模式**：
- **串行**（有依赖）→ 逐个执行，传结果给下一个
- **并行**（无依赖）→ 同时提交，合并结果

**原则**：
- 🧘 **纯执行阶段，不做质量评价和经验评估** — 避免打断心流、浪费 Token
- ✅ 开始前记录目标
- ✅ 完成后记录结果
- ✅ 出错先记录，再尝试恢复或降级
- ✅ **不回调刚给你发消息的 Agent**
- ✅ **临时 Agent 由 helpers/*.ps1 创建，Phase 4 统一回收，不在 Phase 3 内做清理**

---

### 🧹 Phase 4：资源回收与归档

> **这是集中回顾与沉淀的阶段。** 所有阶段性评价、质量总结、经验教训都在这里一次性完成。

1. **生成最终交付摘要**
2. **一次性总评与教训总结**
   - 整体质量评价（非每步打分）
   - 关键经验教训
   - 可复用的模式或方法
3. **状态归档** → `memory/YYYY-MM-DD.md`
4. **清理中间产物** → 问用户：销毁 / 冷存档 / 保留
5. **更新长期记忆** → 有价值的内容更新 `MEMORY.md`
